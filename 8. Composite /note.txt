Composite Pattern Summary

Use the Composite Pattern when objects form a hierarchical tree structure (parent-child relationships), where:

a single object and
a group of objects

should be treated the same way through a common API.

Best Fit Cases : 
File and Folder systems
Organization charts
Comments and replies
Shape groups
Core Idea

A container object can hold child objects of the same abstraction, and those children can contain more children recursively.

Root
├── Item A
├── Item B
│   ├── Child X
│   └── Child Y
└── Item C

Benefits
Simplifies recursive structures
Uniform handling of single and grouped objects
Cleaner client code
Easy to extend
Not Suitable For

Use regular composition/association instead when objects are only related, but not hierarchical.

Examples:

User → Address
Order → Payment
Car → Engine

These are relationships, but not recursive tree structures.

One-Line Rule
Use Composite when:

An object can contain objects of the same abstraction, recursively.

Objects can use other objects via composition
Some composed and singular objects need similar/identical behaviors
Composite design pattern lets us treat both types of objects uniformly
Iteration supported with the Iterator design pattern


Children only = nested data

Children + shared behavior = Composite candidate

Why Behavior Matters

Because the purpose of Composite is:

Treat individual objects and groups of objects uniformly.

Without shared behavior, there is no real reason to use the pattern

One-Line Summary

Composite = recursive children + shared behavior/interface
